### 2020-05-08

#### メモリリーク問題

- メモリリークの直接の原因は、ダングリングポインタを new した後に delete していないことです。
- update current position の後、map kit が prepareGuidanceData を呼び出し、現在位置に最も近い guidance data を見つけようとします。
- prepareGuidanceData は NavInfoProviderImpl::getTrafficLights と NavInfoProviderImpl::getCarParks を呼び出します。
- NavInfoProviderImpl::getTrafficLights を例にとると、呼び出し時に NaviEventOnPath 内のデータポインタを new しています。
- しかし、delete されていません。
- DestEvent にもこの問題があることが判明しました。

解決策：ダングリングポインタを許容しないことが核心です。NaviEventProvider のリファクタリングを決定しました。

- まず、NaviInfoProvider に traffic_light_events と car_park_events の 2 つのフィールドを追加します。
- NaviInfoGenerator で、ルート更新後にこれらの 2 つのフィールドを更新します。
- その後、get のたびに現在の車両位置に基づいてフィルタリングを行います。
- そのため、`PathReader::getAttributes` をリファクタリングする必要があります。以前の実装では現在の車両からの相対オフセットのみを考慮していましたが、Path に対する相対オフセットインターフェースが必要になったためです。

#### PolyLine 修正とテスト

PolyLine の問題は主に、PolyLine 生成時に重複ポイントが入力される可能性があることにあり、それが多くのジオメトリ関連のセグメント計算問題を引き起こします。

1. ベクトル計算：2 つの重複ポイントからゼロベクトルが計算されます。
2. 長さ計算：セグメント長が 0 になり、NaN 問題が発生しやすくなります。

そのため、PolyLine 構築時に PolyLine 内のポイントをチェックし、重複ポイントが見つかった場合は例外をスローすることにしました。

以下の問題が確認されました。

- Jts の getEnd に問題があります。
- Jts の LinearLocation に normalize の問題があります。

  ```scala
  val loc1 = new LinearLocation(0, 1, 1.0)
  val loc2 = new LinearLocation(0, 2, 0.0)

  loc1 compareTo loc2
  // 出力は -1 です。
  ```

- Interpolate 時に、2 つのポイントが近すぎる場合があります。

### 2020-05-13

Bugfix: HDMAPMDK-1211 解決策を思いつきました。

- 根本原因は、プロダクションラインの分割が不合理であることです。

#### 原因分析

車線変更が完了する前に（車線中心線が車線境界線をまたいでいる状態で）、すでに分割が行われていることがわかります。そして、この分割は Rp をまたぐ可能性があります。

lanecenter の形状点に付着した lane border の観測情報は、ジオメトリに忠実に従い、scan line 方式を使用して両側の車線境界線を記録し、意味に基づいてフィルタリングしません。そのため、車線境界線が車線をまたぐ位置では、車線境界線と交差する車線中心線が、その車線境界線を左右両方の lane border ref に同時に記録します。

既存のコードロジックでは、意味に基づいて交差する車線境界線の border ref をフィルタリングする際に、車線変更の種類に基づいて削除する border ref を推測します。

現在のロジックは、このような分割に遭遇すると機能しなくなり、この問題を引き起こします。

#### 解決策

重要なポイントは、**交差する車線境界線の車線変更傾向を見つけること**です。このような分割は Rp をまたぐ可能性があるため、フィルタリングは Rp 単位で行うべきではありません。

1. まず、車線境界線を edge lifting を使用して path に編成します（将来のグラフ再構築のために）。: Seq[LaneCenter]
2. 修正が必要な lane Center と、修正が必要な lane border を見つけます（ここでは、このような lane Center は必ず車線変更によって引き起こされると仮定します）。
3. laneCenter path に基づいて、その LaneCenter の車線変更傾向を計算します。
4. フィルタリングを実行します。

突発的な問題に遭遇した場合、一般的に問題解決の方法は次のようなパターンになることに気づきました。
短期対応策と長期対応策です。問題が露呈する価値は一定であり、時間的制約があるからです。そのため、短期対応策は迅速かつ的確に喫緊の課題を解決することに重点を置きます。では、長期対応策は必要でしょうか？一般的には必要です。なぜなら、問題の特例は、これまで考慮されていなかった解決策の盲点を反映しているからです。この時、問題の原因を分析し、体系的に解決することで、今後同様の問題が適切に解決されるようになり、当初の解決策がより完全なものになります。長期対応策が不要なケースもあります。それは、**徹底的に分析した結果**、この問題を体系的に解決するコストが利益を上回ると判断された場合です（そうだとしても、ほとんどの場合、私たちは主観的には体系的に解決したいと思っています。体系的に問題を解決したいと思わないエンジニアがいるでしょうか？しかし、これには落とし穴があり、重要だが緊急ではないこと、あるいは重要でも緊急でもないことに取り組んでしまい、本来、より価値のある問題を解決するために使える貴重な時間を無駄にしてしまう可能性があります）。

では、ここでの意義をどのように定義するのでしょうか？

現在の仕事の、私にとっての意義は、重要度の高い順に以下の通りです。

1. 問題を提起する能力：価値があり意味のある問題を発見できること。
2. 問題をモデル化する能力を蓄積し、将来、より多様な種類の問題を解決できるようになること。
3. 問題をモデル化することで、一回の努力によって、そのモデルが記述する一種類の問題を解決することを可能にすること。
4. 自分の努力によって、具体的で意味のある問題を解決すること。ここで、一回で一つの問題を解決する意義よりも、一つの努力でこの種類の複数の問題を解決する意義の方が大きい。
5. 楽しさを保つこと、問題解決の成就感、問題そのものの面白さ。
6. 人との交流と関係
7. 給与（これは保障です。重要でないという意味ではなく、他の要素はその上に構築されるべきだという意味です）。

これらが私の仕事の動機のすべてではないかもしれませんが、少なくとも一部の事実を反映しています。

そして、今日私が落ち込んでいる主な理由は、面白さが失われたことにあると思います。まず、問題自体の意義が私には感じられず、やる気を削がれます。第二に、この種の問題のモデル化が不十分であり、その結果、解決手段がすべての問題を解決できない（もちろん、すべての問題を解決できるモデルはほとんど存在しません。モデル自体が問題の単純化と抽象化だからです。ここでは、許容範囲内の問題について議論しています）。これにより、問題が繰り返し発生します。具体的には、QA 担当者から次々と報告されるバグです。その中には、要件の理解不足やテスト自体の問題であるものもありますが、問題分析自体に費やす時間が、解決プロセス全体の中で最も時間を要する部分を占めています。

問題の認識が不十分だと、解決策が本質的でなくなり、人は無駄な作業に時間を費やすことになります。基本的に、上記の意義の 3、4、5 が満たされず、その結果、私は落ち込みます。
