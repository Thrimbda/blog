### 2020年5月8日

#### メモリリーク問題

- メモリリーク問題の直接的原因は、new したダングリングポインタを delete していないこと
- update current position の後、map kit が prepareGuidanceData を呼び出し、現在位置に最も近い guidance data を見つけることを目的としている
- prepareGuidanceData は NavInfoProviderImpl::getTrafficLights & NavInfoProviderImpl::getCarParks を呼び出す
- NavInfoProviderImpl::getTrafficLights を例にとると、呼び出し時に NaviEventOnPath 内のデータポインタを new している
- しかし delete されていない
- DestEvent にも同様の問題があることが判明

解決策：核心はダングリングポインタを存在させないこと。NaviEventProvider のリファクタリングを決定

- まず NaviInfoProvider に二つのフィールド traffic*light_events* & car*park_events* を追加
- NaviInfoGenerator で、route updated の後にこれらのフィールドを更新
- その後、各 get 呼び出し時に現在の車両位置に基づいてフィルタリング
- そのため `PathReader::getAttributes` をリファクタリングする必要がある。以前の実装は現在の車両からの offset のみを考慮していたが、現在は Path からの Offset インターフェースが必要

### 2020年5月13日

バグ修正: HDMAPMDK-1211 解決策を思いつく

- 根本原因はパイプラインの分割が不合理なため

#### 原因分析

車線変更がまだ終了していない状態（車線中心線が車線境界線を横断している最中）で、既に分割が行われており、かつこの分割は Rp を跨ぐ可能性があることが分かる。

lanecenter 形状点に付随する lane border 観測情報は、忠実に幾何形状に基づき、scan line 方式を使用して両側の車線境界線を記録しており、意味に基づくフィルタリングは行われていない。そのため、車線境界線が車線を横断する位置では、車線境界線と交差する車線中心線が、左右の lane border ref に同時にその車線境界線を記録してしまう。

既存のコードロジックでは、交差する車線境界線の border ref を意味に基づいてフィルタリングする際に、車線変更タイプに基づいて削除すべき border ref を推論する。

しかし、現在のロジックはこの種の分割に遭遇すると機能せず、この問題を引き起こす。

#### 解決策

核心は**交差する車線境界線の車線変更傾向を見つける**こと。この種の分割は rp を跨ぐ可能性があるため、フィルタリングは rp 単位で行うべきではない。

1. まず、車線境界線を edge lifting を使用して path に組織化する（将来の graph リファクタリングを考慮）: Seq[LaneCenter]
2. 修正が必要な lane Center と、修正が必要な lane border を見つける（ここでは、この種の lane Center は必ず車線変更によるものと仮定）
3. laneCenter path に基づいて、その LaneCenter の車線変更傾向を計算
4. フィルタリングを実施

突発的な問題に遭遇した場合、一般的に問題解決の方法はこのパターンになるはず：
短期案と長期案。問題が露呈する価値は一定であり、かつ時限性がある。そのため短期案は迅速かつ正確に当面の急務を解決することに焦点を当てる。では長期案は必要か？一般的には必要だ。なぜなら、一つの問題の特例は、以前考慮されていなかった解決策の盲点を反映しているからだ。この時、問題発生の原因を分析し、体系的に解決することで、この種の問題が今後も適切に解決されるようにし、元の解決策をより完全なものにする。長期案が不要な場合もある。それは、この問題を**徹底的に分析した後**、この問題を体系的に解決するコストが利益を上回ると判断した場合だ。（たとえそうであっても、ほとんどの場合、主観的には体系的に解決したいと思う。エンジニアとして、誰が体系的に問題を解決したくないだろうか？しかし、これも罠を残す。人は重要だが緊急でないこと、あるいは重要でも緊急でもないことを解決するために時間を費やし、他のより価値のある問題を解決するために使える貴重な時間を浪費する可能性がある。）
