# Legion の成長軌跡：タスク記録ツールからマルチエージェント工学オペレーティングシステムへ

## はじめに

この文書は、一つの具体的な問いに答えようとするものです：私たちは「最初はほとんど何もない状態」から、どのようにして Legion を現在の、比較的成熟したマルチエージェント工学コラボレーションシステムとして使いこなすようになったのか？

ここで用いる材料は主に以下の三つからなります：

- `.legion/` の git 履歴
- `.legion/config.json` と `.legion/ledger.csv` の状態および監査記録
- ブログ記事『[AI エージェントについての考察](https://0xc1.space/blog/dui-yu-ai-agent-de-si-kao/)』に既に明記されている思考の枠組み

この軌跡を一言で要約すると、おおよそ次のようになるでしょう：

> 私たちは最初、単にエージェントに多くの作業を任せたいと考えていました。しかし後に、本当に構築すべきなのは「より強力なコーディング能力」ではなく、「エージェントの邪魔を減らし、生産性を高め、検証可能で、引き継ぎ可能な」工学システム全体であることに気づきました。Legion はこの過程で、タスク記録ツールから、一歩一歩、マルチエージェントの工学オペレーティングシステムへと成長したのです。

---

## 一、まず現在：Legion はすでに何であるか

始点からではなく、まず現在の形態を見てみましょう。
リポジトリの現状から見ると、Legion はもはや断片的な文書ではなく、継続的に稼働するワークフローシステムとなっています：

- `.legion/config.json` には 33 のタスクエントリが存在します。
- 状態の分布はおおよそ：9 が `archived`、22 が `paused`、2 が `active` です。
- `.legion/ledger.csv` には累計 2339 件の監査記録があります。
- 高頻度のアクションには以下が含まれます：`legion_update_context`、`legion_update_tasks`、`legion_get_status`、`legion_read_context`、`legion_list_reviews`、`legion_respond_review`。
- 現在のタスク作成ポリシーは `agent-with-approval` であり、つまりエージェントはまず提案を行い、明示的な承認なしに直接タスクを作成することはできません。

これらの事実を合わせると、Legion は現在、以下のような複数の安定した能力を備えていることがわかります：

1. **タスクの永続化**：タスク、コンテキスト、進捗状況はすべて外部に保存され、セッション内でのみ存在するものではありません。
2. **設計ゲート**：複雑なタスクは直接着手できず、まず提案、計画、RFC があり、その後実行段階に入ります。
3. **レビューの閉ループ**：コメントは単なるチャット記録ではなく、状態を持つ構造化されたレビュー項目です。
4. **証跡となる成果物**：多くのタスクには `plan/context/tasks` だけでなく、`rfc`、`review-code`、`review-security`、`report-walkthrough`、`pr-body` も生成されるようになっています。
5. **監査とガバナンス**：システムは、誰がいつ、どのような決定を下し、どの段階を進め、どのレビューに応答したかを把握しています。

これは、ブログ記事で述べられた以下のパイプラインと高度に一致しています：

`Intent -> Plan -> Execute -> Verify -> Report -> Memory`

つまり、今日の Legion は「コーディングを補助するメモシステム」ではなく、「複数のエージェントがどのように作業するかを制約するプロトコル層」なのです。

## 二、成長の始点：まずタスクを頭の中から外に出す

Legion の始点は非常に素朴なものでした。

git の履歴を見ると、`.legion` が初めて大規模にリポジトリに登場したのは 2025-12-15 の `implement-quote-service` です。このコミットでは同時に以下が導入されました：

- `.legion/config.json`
- `.legion/ledger.csv`
- `.legion/tasks/implement-quote-service/plan.md`
- `.legion/tasks/implement-quote-service/context.md`
- `.legion/tasks/implement-quote-service/tasks.md`

この始点は重要です。なぜなら、Legion の最小限の三つの構成要素を確立したからです：

### 1. `plan.md`

「何をするか」に答える役割を担います。

`implement-quote-service` では、`plan.md` は単なる ToDo ではなく、比較的完全に以下を記述していました：

- 目標
- 背景と動機
- 非目標
- 範囲
- 段階的計画
- 契約要約
- ベンダー別設計

つまり、最初から「Todo リスト」ではなく、軽量な設計文書だったのです。

### 2. `context.md`

「何が起こったか、なぜそのようにしたか」に答える役割を担います。

この段階では、`context.md` の最も明らかな価値は以下でした：

- 重要なファイルの記録
- 重要な決定の記録
- 検証結果の記録
- 迅速な引き継ぎの提供

本質的には、「頭の中でちょうど理解したこと」を置き換えるものです。

### 3. `tasks.md`

「現在どこまで進んでいるか」に答える役割を担います。

このステップは重要です。なぜなら、マルチエージェントや複数ラウンドのセッションが始まると、最初に失われるのはコードではなく、進捗状況だからです。

ブログ記事では、人間は一日に安定して維持できるコンテキストは二、三つまでであり、それを超えるとスケジューリングが機能しなくなると述べられています。Legion の第一歩は、本質的に「タスクの状態」を人間の頭脳からアンロードすることでした。

したがって、始点段階の Legion は次のように理解できます：

> まずタスクを失わないようにし、コンテキストを回復可能にし、エージェントが作業を終えたら忘れてしまわないようにする。

これはブログ記事の「コンテキストを頭の中からアンロードする」という要求と完全に一致しています。

## 三、第一段階の進化：タスク記録から、暗黙知の外在化へ

第一段階が「今何をしているかを忘れない」ことを解決したとすれば、第二段階が解決したのは、ブログ記事で述べられた「暗黙知の壁」です。

### 1. タスクが複雑化し、レビューが大量に出現し始める

2025年12月の中旬から下旬にかけて、Legion の plan 文書には大量の `> [REVIEW]` ブロックが現れ始めました。

代表的なタスクには以下が含まれます：

- `yuantsexchange-ohlcinterestrate`
- `vendors-ingest-ohlc-interest-rate`
- `task-vex-queryquotes-swr`
- `task-vex-quote-upstream-refactor`
- `vex-series-data-ohlcinterestrate-ohlc-v2`

これらのタスクに共通する特徴は：

- 単に「テンプレートを写す」だけでは完了できない。
- 古い実装、進化の歴史、スタイルの好み、局所的なベストプラクティスが関わっている。
- 成功を決定づけるのは、文法能力ではなく、「何を継承すべきか、何を継承すべきでないかを知っている」こと。

例えば、`vex-series-data-ohlcinterestrate-ohlc-v2` のレビューでは、典型的な「暗黙知の顕在化」の過程が見られます：

- この部分は既存の実装を参考にすべきではない。なぜなら既存の実装には問題があるから。
- 最初は VEX に複雑なレート制限は不要だと考えていたが、後で再判断し、特定のレート制限戦略は保持すべきだと考えた。
- series_id のエンコーディングルール、マージの意味論、キューの公平性などは、モデルがコードから自然に推論できる「世界の知識」ではなく、プロジェクト内部で進化してきた局所的な知識である。

これはまさにブログ記事の階層化に対応しています：

- 第1層は今回のタスク指示。
- 第2層はプロジェクトの技術的決定と局所的なベストプラクティス。
- エージェントが最も失敗しやすいのは、まさに第2層。

Legion はこの段階で、第2層を強制的に書き出す役割を担いました。

### 2. コメントは単なるコメントではなく、コラボレーションインターフェースへ

この段階にはもう一つ重要な変化がありました：コメントは「一時的なつぶやき」から、徐々に閉ループ可能な構造化された入力へと変わり始めました。

例えば：

- あるレビューは設計方針を直接変更するもの。
- あるものは失敗時の意味論を補完するよう要求するもの。
- あるものは過剰設計を取り除くよう要求するもの。
- あるものは既存の誤った実装を参考にしないよう要求するもの。

これらの内容がチャット記録にしか存在しない場合、次のラウンドのエージェントはほぼ確実に失ってしまいます。しかし、`plan.md` や `context.md` に落とし込まれると、「口頭の知識」から「タスクの真実の一部」へと変わります。

したがって、第二段階の Legion は、本質的にもはや単なるタスクトラッカーではなく、**外部化された脳** の機能を担い始めました。

このステップは非常に重要です。なぜなら、ブログ記事で強調された核心的な転換点に対応しているからです：

> 外部化された脳は錦上添花ではなく、複雑なプロジェクトにおける必需品である。

---

## 四、第二段階の進化：外部化された脳から、設計ゲートへ

タスクがますます複雑になるにつれ、単に「知識を書き留める」だけでは不十分になりました。新しい問題は次のようになりました：

> もし複数のエージェントが一緒に動き出しても、方向性自体が間違っていたらどうするか？

この時点で Legion は第三段階に入り始めました：「記録システム」から「設計ゲートシステム」へのアップグレードです。

### 1. RFC と仕様がメインプロセスに入り始める

重要な転換点の一つは `http-proxy-service` です。

このタスクでは、Legion は明らかに「まず実行し、後で記録する」のではなく、「まず設計し、まずレビューし、まずゲートを通し、それから実行する」ようになっていました。

このタスクでは以下が現れました：

- `docs/rfc.md`
- `docs/spec-dev.md`
- `docs/spec-test.md`
- `docs/spec-bench.md`
- `docs/spec-obs.md`
- レビューの結論
- セキュリティレビューのブロック項目
- ウォークスルーと PR 本文

これは、Legion が複雑なタスクをいくつかの安定した階層に分解し始めたことを意味します：

1. **RFC**：意図の整合
2. **Dev/Test/Bench/Obs 仕様**：「どのように検証するか」を事前に明確にする
3. **レビュー**：作業開始前に方向性の問題を可能な限り明らかにする
4. **実装と検証**：低コストのチェックを前倒しする
5. **報告と PR 成果物**：受け入れに役立つ

これはブログ記事で述べられた「意図の整合 + 階層化された検証」と高度に一致しています。

### 2. セキュリティとリソースの問題が、事前のブロックとして現れ始める

`http-proxy-service` や `http-proxy-app-implementation` のようなタスクでは、レビューは単なる提案ではなく、直接 `blocking` が現れるようになりました。

例えば：

- SSRF リスク
- DoS リスク
- 応答ボディサイズの制限がない
- 並行性とキューイングのパラメータがデフォルトで安全ではない
- 環境変数設定の境界が不明確

これは、Legion が新しい役割を担い始めたことを示しています：

> 「なぜそのようにするか」を記録するだけでなく、「なぜ今はできないのか、これらの条件をまず満たさない限り」も記録する。

これが設計ゲートです。

ブログ記事では、マルチエージェントシステムが失敗し始めると、人間の第一反応はより長い RFC を書き、より厳格なレビューを行い、リスクを可能な限り前に押し出すことだと述べられています。Legion はこの時期、まさにこの本能的な反応を制度化したのです。

### 3. それは小さな生産ラインのように機能し始める

この段階まで来ると、Legion の役割はもはや「思い出すのを助ける」ことではなく、「作業の順序を制約するのを助ける」ことになりました。

ブログ記事のパイプラインに対応します：

- `Intent`：ユーザーの目標、非目標、制約
- `Plan`：タスク分解、マイルストーン、境界
- `Execute`：実装
- `Verify`：ビルド / テスト / ベンチマーク / レビュー
- `Report`：ウォークスルー / PR 本文
- `Memory`：コンテキスト / 決定ログ / アーカイブされたタスク

このステップは重要です。なぜなら、次のことを意味するからです：

> Legion はもはやコンテキストを保持するだけではなく、プロセスを保持し始めた。

## 五、マイルストーン：HTTP Proxy シリーズが Legion を真に工学的なものに

これまでの段階にはまだ「やりながら成長する」実験的な感覚があったとすれば、`http-proxy` 関連のタスクは、Legion が真に成熟した最初のマイルストーンと言えるでしょう。

これはブログ記事での私の記述と一致しています：複数のプロジェクトにまたがる `http-proxy` タスクは、確かに「私は基本的にコーディングから離れ、少量のレビューコメントだけを残すことができる」と明らかに感じ始めた転換点でした。

`.legion` を見ると、この判断には十分な証拠があります。

### 1. それは単一のタスクではなく、タスクの集合体である

関連するタスクには少なくとも以下が含まれます：

- `http-proxy-service`
- `http-proxy-app-implementation`
- `vendor-http-services-rollout`
- `http-proxy-metrics`
- `http-services-terminalinfos-ready`
- `vendor-tokenbucket-proxy-ip`

これは点の需要ではなく、相互に連携する一連の工学的タスクです：

- まず基本ライブラリを作成
- 次にアプリケーション層を作成
- 次にロールアウト
- 次にメトリクスを追加
- 次にルーティング / IP 準備状態を追加
- 最後にベンダー側のレート制限と負荷分散ロジックと連携

つまり、Legion は真の **タスク間、パッケージ間、段階間の進化** をサポートし始めたのです。

### 2. レビューのサイクルが明らかに長くなるが、同時により安定する

特に `http-proxy-app-implementation` は、Legion の成熟度を非常によく示しています：

- 一方でレビューの数は多く、論点も多い。
- 他方、これらの論点はチャット記録に残されるのではなく、RFC の更新、レビューの結論、コンテキストの決定へと変わった。

このタスクでは、非常に典型的な工学的な議論が見られます：

- allowedHosts は実際にリクエスト動作に影響するのか、それともメトリクスにのみ影響するのか？
- absolute-form は唯一サポートすべきパス形式なのか？
- invalid_url、blocked、error の境界はどのように定義すべきか？
- target_host / target_path の高基数リスクはどのように制御すべきか？

これらは「コーディング能力」で直接解決できる問題ではなく、**規範の境界、検証の境界、意味論の境界** の問題です。

Legion のここでの価値は、コーディングを手伝うことではなく、これらの境界を安定させるのを手伝うことです。

### 3. 真に受け入れに使用できる成果物を生成し始める

このステップは、ブログ記事の「報告インターフェースは過小評価されている工学的問題である」という記述にも対応しています。

http-proxy シリーズのタスクでは、Legion は安定して以下を生成しています：

- RFC
- review-rfc
- review-code
- review-security
- report-walkthrough
- pr-body
- spec-test / spec-bench / spec-obs

これは、Legion がもはや「仕事をこなす」ことに満足せず、「仕事を明確に説明し、証拠を添付し、リスクを明言する」ことをサポートし始めたことを示しています。

言い換えれば：

> この時点で Legion は「高効率な実行」だけでなく、「低コストな受け入れ」に真に貢献し始めた。

これもブログ記事の重要な判断の一つです：最も高価なのはトークンではなく、手戻りと注意力の漏れです。

報告インターフェースが工学的でない限り、人間は依然としてエージェントが実際に何をしたかを推測するために多くの労力を費やさなければなりません。Legion はこの時期、明らかにこの問題を積極的に解決し始めていました。

## 六、成熟期：工学的パイプラインから、ガバナンスシステムへ

さらに先を見ると、Legion の成熟は単に「文書が増える」ことではなく、**ガバナンス構造** が固定化し始めたことにも現れています。

### 1. タスク作成が承認ポリシーによって制約され始める

現在の `.legion/config.json` の `taskCreationPolicy` はすでに `agent-with-approval` です。

このステップは象徴的です。

それは、Legion が次の事実を認識し始めたことを意味します：

> すべての複雑なタスクが、エージェント自身がいつ作成し、いつ進めるかを決定すべきではない。

この背後には、ブログ記事で述べられたより深層の問題があります：

- モデルがますます強力になるにつれ、プロセスは権限委譲すべきか？
- 権限委譲する場合、境界はどこか？
- 何がまず人間の承認を経る必要があり、何が自動的に進めることができるか？

Legion が提示する答えは完全な自律ではなく、**制御された自律** です。

つまり：

- エージェントは探索、整理、提案ができる。
- しかし、複雑な作業が正式な実行段階に入る前には、依然として人間の承認が必要。

これは、真の組織における作業メカニズムに非常に近いものです。

### 2. レビューは単なる提案ではなく、状態を持つコラボレーションプロトコルへ

ledger の統計を見ると、`legion_list_reviews` と `legion_respond_review` にはすでに大量の記録が累積されています。

これは、レビューが Legion において付随的な能力ではなく、主要な能力の一つであることを示しています。

さらに重要なのは、それが「コメントを読む」ことではなく、以下を行うことです：

- 未解決項目のリストアップ
- 特定のレビューへの応答
- resolved / wontfix / need-info のマーキング
- レビューの閉ループの確認

これは通常のマークダウン注釈とは非常に異なります。それは「コメント」をテキストから、追跡可能な状態機械へと変えました。

このステップの意義は：

> 人間とエージェントのコミュニケーションは、もはやセッションメッセージではなく、沈殿可能、追跡可能、監査可能なプロトコルアクションとなった。

### 3. それは「問題修正」だけでなく、「リスク受容」も担い始める

成熟したシステムのもう一つの兆候は、「すべてのリスクが解決された」ことではなく、「システムがどのリスクを今解決すべきか、どのリスクを今受容すべきかを区別できる」ことです。

`http-proxy-app-implementation`、`vendor-tokenbucket-proxy-ip` などのタスクでは、すでに以下が見られます：

- 一部のセキュリティ問題はレビュー後に `wontfix` とマークされた。
- 一部のリスクはユーザーが受容したものとして明確に記録された。
- 一部の動作は residual risk として保持され、曖昧に忘れ去られることはなかった。

これは、Legion がもはや単なる「バグ修正ツール」ではなく、工学的決定の現実性を担い始めたことを示しています：

- 一部の問題はすぐに修正しなければならない。
- 一部の問題はまず記録し、後で管理する。
- 一部の問題は現在の環境仮定によってカバーされる。

これがガバナンスです。

## 七、現在の最高成熟度の例：`heavy-rfc`

既存のタスクから Legion の成熟度を最もよく体現する代表を選ぶなら、`heavy-rfc` を選びます。

このタスクは、Legion の現在のワークフローの完全なパラダイムサンプルと見なすことができます。

### 1. 最初からリスク分類と段階宣言がある

単に「live trading を実装する」と書くのではなく、最初から明確に以下を宣言しています：

- `rfcProfile=heavy`
- `stage=design-only`
- `risk=high`

つまり、このタスクはまず実装し後で文書を追加するのではなく、まずそれが高リスクタスクであることを認め、heavy RFC プロセスを経なければならないとしているのです。

### 2. 完全な成果物チェーンがすでにある

`heavy-rfc` の下にはすでに以下が含まれています：

- `task-brief.md`
- `research.md`
- `rfc.md`
- `review-rfc.md`
- `test-report.md`
- `review-code.md`
- `review-security.md`
- `report-walkthrough.md`
- `pr-body.md`

この成果物チェーン自体が、Legion が高リスクの工学的タスクを、レビュー可能、検証可能、引き継ぎ可能な複数の階層に分解したことを示しています。

### 3. ブログ記事の「まず意図を収束させ、それから実行を開放する」作業方法を体現している

ブログ記事には非常に核心的な判断があります：

> 自律性は賢さだけで十分なのではなく、邪魔が少なく、生産性が高く、検証可能であること。

`heavy-rfc` はまさにこれを体現しています：

- まず設計文書とレビューで方向性を確定させる。
- 次にテストとレビューで実装リスクを低減する。
- 最後に report / PR body で受け入れコストを下げる。

これは、Legion が現在、新しい作業姿勢をサポートできるようになったことを意味します：

> 人間は主に目標設定、境界制約、重要なレビューを行う。エージェントは制度化された軌道内で実装、検証、報告を推進する責任を負う。

これはまさにブログ記事の最後で要約された役割の転換です：実行者から、レビュアー、意思決定者、システム反復者へ。

## 八、タスクの「程度と範囲」から見る成熟度の向上

すべてのタスクを一緒に見ると、Legion の成熟は単にプロセスの複雑さが増すだけでなく、タスクタイプ自体が向上していることがわかります。

### 1. 初期：単点実装型タスク

代表：`implement-quote-service`

特徴：

- 単一テーマ
- 境界が比較的明確
- 文書は主に理解と引き継ぎに役立つ
- Legion は主にタスクトラッキングを担当

### 2. 中期：設計論争型タスク

代表：`vex-series-data-ohlcinterestrate-ohlc-v2`、`yuantsexchange-ohlcinterestrate`

特徴：

- 複数のモジュールにまたがる
- 大量の歴史的負債と局所的なベストプラクティスが関わる
- コメント / レビューが密集
- Legion は主に暗黙知の外在化を担当

### 3. マイルストーン期：プロジェクト横断工学的タスク

代表：`http-proxy-*` シリーズ

特徴：

- パッケージを横断
- RFC / 仕様 / ベンチマーク / セキュリティレビューがある
- ロールアウト、観測、ロールバック、報告がある
- Legion は主に完全な工学的パイプラインを担当

### 4. 現在：高リスクガバナンス型タスク

代表：`heavy-rfc`

特徴：

- リスク分類が明確
- 承認ゲートが明確
- レビューの閉ループが明確
- 文書と証跡チェーンが完全
- Legion は主にガバナンスと納品プロトコルを担当

言い換えれば、タスクの「程度と範囲」自体が、Legion の成熟度を映す鏡なのです。

最初は「一つの機能を作る」ことしか処理できませんでした。その後、「一連の複雑な工学作業を行う」ことを処理し始めました。現在では、「高リスク作業を制御された方法でどのように推進するか」を処理しています。

## 九、この軌跡がブログ記事の考察にどのように対応するか

『AI エージェントについての考察』を振り返ってみると、ブログ記事に書かれた多くの判断が、すでに Legion の歴史の中で実現されていることがわかります。

### 1. 「初めて scale の甘い点を味わう」

ブログ記事では：複数のエージェントが並行してタスクを推進すると、短時間で機械的な収穫の快感を味わうことができる。

Legion の歴史では、2025年12月にタスクが急速に増加した時期に対応します：

- quote service
- quote routing
- SWR
- scheduler
- OHLC / interest rate
- token bucket

これは、最初の核心目標が確かに **まずエージェントに多くの作業を任せる** ことだったことを示しています。

### 2. 「ボトルネックは私自身」

ブログ記事では、実行作業がエージェントに移ると、人間の真のボトルネックはコンテキスト管理、受け入れ、意思決定になると述べられています。

Legion の最初の三つの構成要素は、まさにこの問題を解決しています：

- `tasks.md` でコンテキストの喪失を低減
- `context.md` で決定と重要なファイルを記録
- `plan.md` でタスク目標が漂流しないようにする

### 3. 「暗黙知の壁」

ブログ記事では、エージェントが学ぶのはしばしば可視サンプルであり、現在の標準が何であるかを知らないと述べられています。

Legion の対応は：

- レビューを plan に書き込む
- 制約を context に書き込む
- 設計論争を構造化文書として書く

つまり：暗黙知を外在化する。

### 4. 「意図の整合 + 階層化された検証」

ブログ記事のパイプラインは、`http-proxy` と `heavy-rfc` でほぼそのまま実現されています：

- Intent：Goal / Non-goals / Scope
- Plan：Phase / RFC / Design Summary
- Execute：実装
- Verify：test / review-code / review-security / bench
- Report：walkthrough / pr-body
- Memory：context / archived task / ledger

### 5. 「報告インターフェースは過小評価されている工学的問題」

ブログ記事は、結論は可能な限り成果物に結びつけるべきだと強調しています。

Legion の実践は明らかにこの方向に向かっています：

- 結論は一言ではなく、report、review、test-report、PR body に対応する。
- 人間はすべてのコードを再読する必要はなく、まず凝縮された成果物を読むことができる。

まだ私が想定している Citation Agent ではありませんが、方向性はすでに明確です。

### 6. 「ベンチマークは必需品になる」

ブログ記事では、将来は感覚ではなく、異なるワークフローやモデルバージョンを比較できなければならないと述べられています。

これは Legion ですでに初期実装が現れています：

- `spec-bench.md`
- ベンチマークシナリオと閾値
- ベンチ出力と報告

つまり、この道はもはやアイデアではなく、工学的になり始めているのです。

## 十、最も重要な変化：Legion が変えたのはエージェントだけでなく、人間の役割も

表面的に見ると、Legion の成長は次のように見えるかもしれません：

- 文書が増えた
- レビューが増えた
- プロセスが長くなった

しかし、真に重要な変化はこれらではなく、**人と機械の役割分担が再定義された** ことです。

より早い時期には、人間の役割はおおよそ次のようなものでした：

- 自ら実行する
- 自ら記憶する
- すべてのコンテキストを自らカバーする

Legion が徐々に成熟するにつれ、人間の役割は徐々に次のようになりました：

- 目標と制約を設定する
- 設計の境界を審査する
- ブロックレベルのレビューを処理する
- 成果物とリスクを受け入れる
- コラボレーションシステム全体を反復する

これがブログ記事の最後で要約された言葉です：

> 私が今行っているのは「AI でより多くのコードを書く」ことではなく、「AI で自分自身をスケールする」ことである。

Legion はまさにこの目標の工学的実装です。

それは「自分自身をスケールする」ことを抽象的な願望から、実装可能、監査可能、復習可能、継続的に最適化可能なコラボレーション構造へと変えました。

## 十一、現在から次のステップを見る：`~/Work/legion-mind` が表す最新の進化方向

Yuan リポジトリの `.legion/` の歴史だけを見ると、Legion が実際のプロジェクトでどのように少しずつ追い出されてきたかがわかります。しかし、現在の `~/Work/legion-mind` を見ると、Legion がすでに次の段階へ進化し始めていることがわかります。

このステップは非常に重要です。なぜなら、Legion の目標がもはや「一つのリポジトリで使いやすい」ことだけでなく、この経験を **インストール可能、移行可能、ベンチマーク可能、再利用可能** な汎用 Autopilot Kit に精錬しようとし始めていることを示しているからです。

`~/Work/legion-mind/README.md` と `docs/legionmind-usage.md` から、最新の方向には少なくとも五つの特徴があることがわかります。

### 1. 「リポジトリ内ワークフロー」から「汎用オーケストレーションテンプレート」へ

Yuan では、Legion は最初は具体的なタスクに沿って成長しました。`legion-mind` に至って、それは明示的に汎用エージェントオーケストレーションテンプレートとして抽象化されています：

- primary agent：`legion`
- subagents：`engineer`、`spec-rfc`、`review-rfc`、`review-code`、`review-security`、`run-tests`、`report-walkthrough`
- skill：`skills/legionmind`

これは、最新バージョンの Legion が「マルチエージェントコラボレーション」を一時的に数人のエージェントを集めて作業することと理解するのではなく、次のようにモデル化し始めたことを示しています：

- orchestrator はプロセスの推進を担当
- subagent は単一の責任を担当
- `.legion/` は永続化状態と監査を担当

つまり、Legion は「経験型ワークフロー」から「役割が明確なオーケストレーションシステム」へと変わりつつあるのです。

### 2. 手動呼び出しからコマンド化されたエントリーポイントへ

`legion-mind` で最も明らかな変化の一つは、一般的なワークフローがコマンド化されたことです：

- `/legion`
- `/legion-impl`
- `/legion-rfc-heavy`
- `/legion-pr`
- `/legion-bootstrap`
- `/evolve`

これは一見、使用体験の最適化に過ぎないように見えますが、本質的にはそうではありません。

それは、Legion が「どの段階を、どの順序で実行するか、いつ設計のみを行うか、いつ実装を続けるか、いつ経験を沈殿させるか」といった約束事を、
